Why Pi

The substrate matters.

ralph - works is built on Pi because the agent platform matches the way this workflow needs to evolve: actively maintained, quickly adopted by developers, lightweight, customizable, open source, and model agnostic.

Maintained A live platform gives the extension room to keep improving with the dev community.
Lightweight The workflow can stay close to the repo instead of becoming a heavy orchestration layer.
Customizable Commands, gates, skills, checkpoints, and artifacts can be shaped around the project.
Open source The control surface is inspectable, forkable, and easy to adapt for different teams.

Key takeaway

Autonomy and control do not have to conflict.

ralph - works is built around maximizing both in a safe way. The agent gets room to work inside each phase; the developer gets visible boundaries, artifacts, checkpoints, and review paths.

Autonomy Ralph can plan, attack, harden, implement, and review.
Control The route, gates, artifacts, and handoffs stay explicit.
Safety Defaults pause before TDD; review can route work back.
01

Persistence

The loop is the primitive.

Standard Ralph is useful because it keeps an agent moving through work, completion, reflection, and saved state. That persistence is the foundation.

02

Problem

The weak point is the missing map.

If everything is another iteration, planning, threat modeling, implementation, and review can blur into one long turn. The agent keeps moving, but the operator has less to inspect.

03

Route

ralph - works gives the run named phases.

The work becomes legible: spec, red team, harden, optional HTML, TDD implementation, and review. Each phase has a job and an expected output.

04

Control surface

The default checkpoint is the handoff.

Ralph drafts, attacks, and hardens the plan. Then it stops. The engineer can continue, render HTML first, or use --yolo to skip human review stages and run straight through the entire loop without checking in.

05

Evidence

The run leaves material engineers can inspect.

The persuasive part is not only the terminal. It is the growing evidence trail: spec, red-team findings, hardened changelog, optional HTML, and live status.

06

Review loop

Tests passing is not the only finish line.

Review can send critical findings back to implementation. LGTM completes the run. The last phase is iterative by design.

.ralph/

Model plan JSON

The model choice becomes part of the architecture.

.ralph/model-plan.json lets a team declare which model should carry the run, or which phases deserve a different reasoning level. It is intentionally trusted only when the operator opts in.

Explicit Model routing is visible in the repo instead of hidden in a prompt.
Phase-aware Planning, implementation, and review can each use a purpose-fit model.
Trusted Workspace config is ignored unless started with --trust-model-plan.

Gates JSON

The pass/fail policy is explicit, not guessed.

.ralph/gate-config.json tells ralph - works which checks it is allowed to run. If the file is absent, the agent must use the repository's documented test commands manually.

Opt-in Ralph-managed gates run only when this file exists.
Safe Commands are whitelist-checked and shell metacharacters are rejected.
Useful Passing implementation gates can hand the run into review.

Operating mode

The operator chooses the handoff.

Default mode pauses before TDD. Yolo mode skips every human checkpoint and sends the agent straight through the full loop without asking for review.

Default mode

Pause before implementation.

Ralph generates, attacks, and hardens the plan, then waits for the engineer to approve the implementation handoff.

Operating truth

Adoption is easier when the limits are visible.

The page should sell the improvement without hiding the engineering constraints that matter in real projects.

Skills are required

/ralph-works start validates selected phase skill files before launch.

HTML is opt-in

Rendering runs only with --render-html, html, or an explicit render phase.

Gates are explicit

Ralph-managed gates come from .ralph/gate-config.json; the extension does not guess.

Use a worktree

Implementation belongs in an isolated worktree so the primary checkout stays available.

Close with action

Standard Ralph keeps going. ralph - works knows where it is going.

Install the extension, reload Pi, and start with one bounded feature. The page should leave the reader with a single command.
