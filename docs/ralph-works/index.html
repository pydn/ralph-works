Pi extension for long agent runs

ralph - works

AI agents write fast. Sometimes too fast: no spec, no threat model, tests after the fact, and a review that arrives when the assumptions are already baked in.

ralph-works is the boring engineering answer: give the agent a disciplined loop, make each handoff visible, and let the developer keep the final say.

Show the shape first

The loop is not the problem. The missing map is.

The Ralph Wiggum methodology gives an agent persistence. ralph-works keeps that persistence, then adds a visible route through planning, pressure, implementation, and review.

/ralph-works start 6 named phases
01
Spec Define scope, architecture, and tradeoffs before code exists.
02
Red team Attack the plan early, while the fix is still cheap.
03
Harden Patch the spec, not just the implementation.
04
HTML Optionally render a readable artifact for review.
05
TDD Write failing tests, implement, then run configured gates.
06
Review Loop on critical findings until the run earns LGTM.

Standard Ralph

Useful persistence.

The agent keeps going: work an item, call a completion tool, reflect on what remains, resume from state. That is a good primitive. It is also easy to turn into one very long "trust me" conversation.

ralph-works

Persistence with receipts.

The agent still gets autonomy inside each phase. The developer gets boundaries: artifacts to inspect, gates to configure, checkpoints to approve, and a review loop that can send work back instead of waving it through.

Why Pi

The substrate matters.

Pi is the right home for this because it is actively maintained, rapidly adopted by developers, lightweight, highly customizable, open source, and model agnostic. The last point is the big one.

Maintained

A live agent platform lets the extension improve with the ecosystem instead of freezing around one workflow.

Lightweight

The pipeline stays close to the repo. No heavyweight orchestration service is required to teach the loop discipline.

Customizable

Teams can shape commands, gates, skills, checkpoints, and artifacts around their actual codebase.

Open source

The control surface is inspectable and forkable. That matters when the workflow is allowed to edit code.

Configuration as a teaching aid

The important decisions live in files.

Model routing and quality gates are not hidden in a prompt. They sit in the repo where engineers can read, diff, review, and change them.

.ralph/model-plan.json routing 
{
  "planning": {
    "provider": "anthropic",
    "model": "opus-4.7",
    "thinkingLevel": "xhigh"
  },
  "implementation": {
    "provider": "qwen",
    "model": "qwen3.6-27b-mtp"
  },
  "review": {
    "provider": "openai",
    "model": "gpt-5.5",
    "thinkingLevel": "xhigh"
  }
}
.ralph/gate-config.json quality gates 
{
  "version": "1.0",
  "language": "typescript",
  "gates": [
    {
      "name": "Type Check",
      "command": "npx tsc --noEmit"
    },
    {
      "name": "Tests",
      "command": "npx vitest run"
    }
  ]
}

Slowing the agent down is not a retreat from autonomy. It is how you make autonomy useful on code you actually have to maintain.

The agent explores, writes, tests, and remediates. The developer chooses the route, inspects the evidence, and keeps the final say. Apparently both can be true. Useful, that.

Try the boring version first

One command starts the run.

Default mode pauses before implementation so the planning artifacts can be reviewed. Use --yolo only when you deliberately want to skip human review stages and run straight through.

/ralph-works start "add refresh-token revocation"
